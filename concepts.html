CONCEPTS

ITEM

 Something able to be used for loading the storage with it (data object, bucket, etc).

  • Data item

    The tool is designed to perform I/O operations on abstract data items. The data item may be a:

    • block/chunk/range of data;
    • data object having some kind of identifier;
    • file.

  • Container item

    S3 Bucket, Swift container or anything else.

  • Token item

    Swift auth token, Atmos subtenant or anything else.

ITEMS INPUT

 The entity which may used for obtaining the items from an underlying source. An items input also may be:
  • items collection in the memory
  • file
  • storage container listing
  • new items generator
  • etc

ITEMS OUTPUT

 The entity which may be used as an items output. Any load executor also is an items destination. Putting an item to a load executor means submitting the execution of the I/O operation defined by the load job using this item. An items output also may be:
  • items collection in the memory
  • file
  • etc

LOAD JOB

 A particular set of items linked with some kind of I/O operation (create/read/update/delete).

LOAD EXECUTOR

 The entity which executes a load job. May have specified properties such as number of concurrent connections. Any load executor may be also a items output and always use some kind of external items input.

DISTRIBUTED MODE

  • Load client (formerly "Controller")

    A tool process which interacts with other tool processes via the network (distributed mode) which doesn't work with a storage directly and only configures/starts/monitors/stops a distributed load execution. Also a load client fetches and aggregates the results. A load client may use many load servers to perform distributed load execution.

  • Load server (formerly "Driver")

    A tool process which interacts with other tool processes via the network (distributed mode) which performs actual load execution requested by a load client. A load server process may be reused many times by different load clients w/o restart but maximum one load client may use the load server at every moment of time. The concurrent usage of the same load server by different load clients is an invalid way to do the things.

METRICS

 Mongoose reports the following metrics:

  • Count

    The number of successful operations and the number of failures, and number of operations that are being performed.

    Note that Mongoose never retries failed operations./p>

  • Duration

    The time to perform an operation (reported since version 1.0). This is the time between the moment the first byte of a request was sent and the moment the first byte of a response was received.

    • A - the moment the first byte of a request was sent
    • B - the moment the last byte of a request was sent
    • C1 - the moment the response was received
    • C2 - the moment the first byte of the response data was received
    • D - the moment the last byte of a response was received
    • duration = D - A

  • Latency

    The response time. Time between the moment the last byte of a request was sent and the moment the first byte of a response was received. See the picture above, the latency is C1-B

  • Throughput (TP)

    The item processing rate - the number of the items per second.

  • Bandwidth (BW)

    The data processing rate measured in megabytes per second.

Note Mongoose reports effective duration, latency, throughput, and bandwidth. When writing a huge data item, the time based metrics reported will be zero until the first data item is completely written. These metrics will also be zero when all operations fail.
